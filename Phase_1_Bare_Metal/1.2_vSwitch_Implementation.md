# Phase 1, Step 1.2: Isolated Host-Only vSwitch Configuration

**Date Completed:** June 2025  
**Objective:** Create a completely isolated virtual network (10.0.0.0/24) for the lab environment  
**Status:** ✅ Complete

---

## What This Step Does

This step configures VMware Workstation Pro to use a **host-only virtual switch** – a completely isolated network that:

- ✅ Does NOT connect to your home LAN
- ✅ Does NOT connect to the internet
- ✅ Uses static IP assignment (no DHCP)
- ✅ Creates a predictable, repeatable infrastructure topology

All future VMs (Windows Server, Windows 11, Ubuntu) will connect to this isolated network.

---

## Why This Matters

A proper lab environment must be **sandboxed**. If your lab's network traffic leaked into your home network, it could:
- Poison your home DNS
- Interfere with your personal devices
- Create security risks
- Break reproducibility for Infrastructure as Code

This step creates that safety boundary.

---

## Prerequisites

- ✅ VMware Workstation Pro installed
- ✅ Administrator access on your computer
- ✅ Understanding that static IPs enable Infrastructure as Code automation

## Step-by-Step Implementation

### Step 1: Open Virtual Network Editor

1. Launch **VMware Workstation Pro**
2. Click **Edit** → **Virtual Network Editor** (or Ctrl+Shift+N)
3. When prompted for administrator privileges, click **"Change Settings"** to elevate permissions

### Step 2: Select VMnet1 (Host-Only Network)

1. In the Virtual Network Editor window, look at the table showing existing networks
2. You'll see:
   - VMnet0 (Bridged)
   - VMnet1 (Host-only) ← **Select this one**
   - VMnet8 (NAT)

3. **Click on the VMnet1 row** to select it

### Step 3: Configure the Subnet

In the "VMnet Information" section below, ensure:

1. **Host-only** radio button is selected (not Bridged, not NAT)
2. Look at the "Subnet IP:" field
3. **Change it to:** `10.0.0.0`
4. Look at the "Subnet mask:" field
5. Verify it shows: `255.255.255.0` (this is /24 CIDR notation)

### Step 4: Disable DHCP (Critical)

1. Find the checkbox labeled **"Use local DHCP service to distribute IP address to VMs"**
2. **UNCHECK this box** (remove the checkmark)
3. This forces all VMs to use **static, manually-assigned IP addresses**

Why? Because in a lab where you're practicing Infrastructure as Code, predictable IPs are essential. DHCP would reassign addresses unpredictably.

### Step 5: Apply Changes

1. Click **"Apply"** button to save changes
2. Click **"OK"** to close the Virtual Network Editor

## Lessons Learned

### Why Static IPs Matter for IaC

When you later write Terraform code to provision infrastructure, it needs to know:

> "Where will this service live? What IP address will it have?"

With DHCP enabled, the answer would be: "I don't know – it changes every time."

With static IPs: "It will always be at 10.0.0.10 because we configured it that way."

This predictability is **essential** for Infrastructure as Code.

### Why Host-Only, Not NAT or Bridged

- **Bridged:** Would connect VMs directly to your home network – security risk
- **NAT:** Would route through your host's IP – still accessible from home network
- **Host-only:** Completely isolated – only your host can communicate with it
